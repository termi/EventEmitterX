## Human-Readable Summary

### Frontend (demo/eventSignals-test-app)
- **Sub-path deployment support**: The demo app now correctly works when hosted under a non-root path (e.g., `/EventEmitterX/` on GitHub Pages).
  A `<base href>` tag is dynamically injected into `<head>` based on the detected application root, ensuring all (? need more work)
  relative asset and navigation paths resolve correctly.
- **Safari compatibility fix**: The modern `<select>` customization feature (`appearance: base-select`) is now correctly disabled in Safari,
  which can return a false positive for custom select support detection.
- **Dynamic locale list**: The i18n locale selector now automatically adds a newly selected locale to the allowed locales
  list if it was not previously included in the default list.

## Technical Details

### `demo/eventSignals-test-app/lib/history_navigation.ts`
- Added `applicationRoot` export: an IIFE that computes the base path of the app from `globalThis.location.pathname`
  (strips the file name part, ensures trailing slash). Used to set `<base href>` and resolve asset paths.
- Removed an obsolete `@ts-expect-error` comment for `page404` component type.
- Added `import type { JSX } from "react"`.

### `demo/eventSignals-test-app/app.tsx`
- Imports `applicationRoot` from `history_navigation`.
- Before calling `initNavigation`, dynamically inserts `<base href="${applicationRoot}">` into `<head>` when `applicationRoot` is
  non-empty. Fixes asset resolution (images, scripts). todo: Need more work to fix Router navigation under sub-paths.

### `demo/eventSignals-test-app/pages/10.GlobalTimes.tsx`
- Changed `src="/pip.svg"` → `src="pip.svg"` (relative path). The absolute `/pip.svg` was broken under GitHub Pages
  sub-path deployment; the relative path works correctly with the new `<base href>` tag.

### `demo/eventSignals-test-app/lib/utils.ts`
- Added `checkSafariBrowser()` utility function: detects Safari using `navigator.userAgent` and `navigator.vendor`.

### `demo/eventSignals-test-app/lib/supports.ts`
- `hasCustomizableSelectSupport()`: added `&& !checkSafariBrowser()` condition to prevent false positives in Safari (related WebKit bug: [bugzilla #286642](https://bugs.webkit.org/show_bug.cgi?id=286642)).

### `demo/eventSignals-test-app/modules/LocaleSelectorModern.tsx`
- Fixed TypeScript ambient JSX declaration: moved `IntrinsicElements` from deprecated global `namespace JSX` into `namespace React { namespace JSX { ... } }` — correct approach for React 19+ / `@types/react` v19.

### `demo/eventSignals-test-app/modules/NavBar.tsx`
- Added a commented-out TODO block describing the future need to use an `absolutePath` (combining `applicationRoot + routerPath`)
  instead of a bare relative `routerPath` in `<a href>` attributes to prevent broken navigation in sub-path deployments.

### `demo/eventSignals-test-app/state/i18n.ts`
- Extracted the hard-coded locale list into a module-level constant `defaultAllowedLocalesList` (was previously inline inside `_getAllowedLocalesList()`).
- Added a module-level mutable array `customLocalesList: string[]` that **accumulates** non-default locales for the entire session lifetime.
- `currentLocale$` setter: when `newLocale` is absent from both `defaultAllowedLocalesList` and `customLocalesList`, it is pushed into `customLocalesList`.
  Then, if `customLocalesList` is non-empty, `currentLocale$.data.allowedLocales` is updated by calling `_getAllowedLocalesList(customLocalesList)`.
  This ensures the locale selector always includes any previously chosen non-default locale, not just the current one.
- `_getAllowedLocalesList(customLocalesList?)` now accepts an optional array of extra locales to merge with the defaults.
